# 悟道 v2 - 个人交易终端与 AI 协同交易工作台

> 基于原始文档《产品设计.md》补强整理，并结合当前已有 `~/projects/trading-agent/` 原型能力形成的新版本设计文档。

## 1. 产品定位

“悟道”不是一个普通看盘软件，也不是单纯的自动交易脚本。

它的目标是成为一个**面向 Jason 本人的个人交易终端**，同时也是一个**供人类交易者与私人 AI Agent 协同工作的统一交易工作台**。

它要解决的核心问题不是“下单”本身，而是：

1. **把交易行为放进一个可监督、可约束、可复盘的统一系统中**。
2. **让 AI Agent 参与交易研究、信号分析、纪律检查、策略管理和复盘，但不默认拥有无边界执行权**。
3. **把人工交易、规则策略、AI 辅助、交易日志、纪律系统、风控与回测，放进同一个产品闭环。**

一句话定义：

**悟道 = 个人交易终端 + 纪律执行系统 + AI 协同策略工作台。**

---

## 2. 产品目标

### 2.1 核心目标

1. **交易纪律可执行**
   - 在下单前、持仓中、平仓后都能对纪律进行检查与约束。
   - 不是靠“记住规则”，而是靠系统强制执行规则。

2. **人机协同而非黑箱自动化**
   - AI Agent 提供建议、分析、参考资料、策略方案、风险提示。
   - 默认模式下，AI 不应在后台默默推进真实项目或交易执行。
   - 所有涉及真实交易、策略启停、参数变更、下单权限的动作，应有明确边界和授权机制。

3. **统一交易工作流**
   - 从市场观察 → 交易计划 → 信号产生 → 纪律检查 → 下单执行 → 持仓管理 → 复盘归档，全部放到一个系统中。

4. **统一管理多种策略形态**
   - 宏观/新闻策略
   - 链上追踪策略
   - 数据驱动/套利策略
   - 网格策略
   - 特定平台策略（如 Polymarket）

5. **先个人可用，再逐步增强**
   - 第一阶段必须是 Jason 本人每天愿意打开使用的工具。
   - 不追求一开始就做成多用户 SaaS 或全自动量化平台。

### 2.2 非目标

当前阶段不以以下内容为第一优先级：

- 多用户协作
- 公共云多租户架构
- 高频交易
- 自建完整图表引擎
- 自建完整撮合系统
- 从零造券商底层基础设施

原则：**优先做“交易决策与执行工作台”，而不是“从零造一整个券商系统”。**

## 3. 设计原则

### 3.1 本人优先
这是 Jason 自己使用的系统，设计应围绕：
- 自身交易习惯
- 自身风险偏好
- 自身认知漏洞与纪律短板
- 自身与 AI Agent 的协作方式

### 3.2 纪律优先于便利
若“方便下单”和“防止犯错”冲突，优先选择后者。

### 3.3 AI 默认是参谋，不是默认操盘手
系统应支持未来的自动化，但默认模式下：
- AI 负责研究、提示、建议、整理、归纳
- AI 不默认直接执行真实交易
- 真实执行要么手动确认，要么进入严格授权模式

### 3.4 可审计
任何重要动作都需要可回溯：
- 信号从哪来
- 为什么下单
- 哪条纪律放行/拦截了
- 哪个策略版本在运行
- 哪个 Agent 给了什么建议

### 3.5 复用成熟能力
图表、回测、券商 API、TradingView webhook、消息通知等，优先复用成熟方案，不重复造轮子。

### 3.6 研究与执行解耦
悟道不应把研究代码、模型代码、页面逻辑、执行逻辑直接搅成一团。

系统必须允许：
- 研究侧独立产出信号与评估结果
- 执行侧只消费标准化产物
- UI 负责展示、审批、解释，而不是硬编码交易逻辑

这意味着悟道不是一坨会下单的脚本，而是一条**从研究到执行的可追溯流水线**。

### 3.7 策略容器优先于模型崇拜
悟道需要兼容：
- 规则型策略
- 数据驱动策略
- AI / Agent 建议型策略

但不应该被某一种模型栈绑死。

换句话说，悟道应该是**策略容器**，而不是某个 `LSTM`、`Transformer` 或某个量化框架的前端壳子。

## 4. 用户与角色

虽然当前产品面向 Jason 单人使用，但系统内部存在多个“角色”。

### 4.1 人类交易者（Jason）
职责：
- 制定交易计划
- 审批重要策略与参数
- 决定是否执行真实交易
- 审查 AI 建议与回测结果
- 复盘和修正纪律

### 4.2 AI Agent
职责：
- 研究与总结
- 生成交易建议
- 归纳参考资料
- 发现风险
- 协助策略设计、监控、复盘

边界：
- 默认不能擅自执行真实交易
- 默认不能静默修改核心策略与风控参数
- 默认不能在后台暗中推进“真实产品开发”或“交易执行”

### 4.3 执行适配器（TradingView / 券商 / 平台）
职责：
- 接收信号
- 提交订单
- 拉取订单状态 / 成交 / 持仓
- 反馈执行结果

## 5. 产品总体架构

从“产品闭环”视角看，悟道应先被理解为一条四段流水线：

1. **市场数据层（Market Data Layer）**
   - 行情接入、历史数据、watchlist、指标原始输入
2. **策略与信号层（Strategy / Signal Layer）**
   - 规则策略、指标策略、数据驱动策略、AI 候选建议
3. **研究与评估层（Research / Evaluation Layer）**
   - 回测、参数实验、绩效比较、研究结论沉淀
4. **执行与部署层（Execution / Serving Layer）**
   - `Signal` / `OrderIntent` 消费、纪律检查、人工确认、真实执行

这个四层结构描述的是**研究到执行的业务流水线**。

而在系统实现上，悟道仍建议采用下面的六层结构：

### 5.1 展示层（UI / Terminal）
负责：
- 看盘
- 持仓展示
- 策略筛选与管理
- 纪律提示
- 交易确认与拦截
- 复盘查看
- AI 建议面板

### 5.2 应用编排层（Application Layer）
负责：
- 串起交易流程
- 下单前校验
- 策略启停
- 警报调度
- Agent 建议汇总
- 日报/周报生成

### 5.3 纪律与风险控制层（Discipline & Risk Engine）
负责：
- 下单前规则检查
- 仓位上限检查
- 单日亏损限制
- 连续亏损冷静期
- 止盈止损、分批减仓、移动止损、保本线等高级规则

### 5.4 策略与 Agent 协同层（Strategy & Agent Layer）
负责：
- 策略定义
- 策略版本控制
- 参数管理
- 策略适用标的管理
- AI 提案、AI 解释、AI 复盘

### 5.5 数据与执行适配层（Data / Execution Adapters）
负责：
- 行情接入
- TradingView webhook
- 券商/交易所连接
- 新闻/宏观/链上/第三方数据接入
- Polymarket 等专项平台接入

### 5.6 存储与审计层（Storage / Audit Layer）
负责：
- 订单、成交、持仓、信号、策略版本、纪律日志、Agent 建议、复盘结果等持久化
- 审计追踪
- 查询与统计

## 6. 核心功能设计

## 6.1 行情与观察模块

### 目标
为 Jason 提供统一的市场观察入口，支持股票、加密货币、未来扩展到更多资产类别。

### 功能
- 自定义 watchlist
- 股票与 crypto 的统一行情展示
- 多周期 K 线 / 简要图表
- 成交量、涨跌幅、波动性、技术指标摘要
- 自选池分组（核心观察 / 候选交易 / 已持仓 / 策略绑定标的）
- 异常波动提醒

### 结合现有原型能力
当前已有：
- `data_pipeline.py`：股票 + crypto 行情拉取
- `dashboard.py`：基础看盘 Dashboard

可复用为：
- 悟道的初版“行情数据层”
- 初版观察面板的基础能力

不足：
- 仍是零售级数据接入
- 尚无实时流、缓存、统一时钟、异常恢复机制

---

## 6.2 交易执行模块

### 目标
支持人工交易、TradingView 信号转发、以及未来的半自动/自动交易执行。

### 关键输入原则
执行模块不应直接消费杂乱的研究代码输出。

更合理的方式是：
- 研究/策略侧先产出标准化 `Signal Artifact`
- 执行侧将其转换为系统内部 `Signal` 或 `OrderIntent`
- 再交给纪律与风控引擎审查
- 最后才进入人工确认或受控执行

### 功能
1. **TradingView 接入**
   - 接收 webhook 信号
   - 将信号转换为系统内部的 `Signal` 或 `OrderIntent`
   - 绑定策略版本与标的

2. **券商/交易所适配器**
   - 下单
   - 撤单
   - 查询订单状态
   - 查询成交与持仓
   - 查询账户资产

3. **订单管理**
   - 新单 / 撤单 / 修改
   - 条件单
   - 分批进出
   - 分批止盈
   - 追踪止损
   - 保本止损
   - 风险平仓

4. **交易确认流**
   - 下单前展示：
     - 信号来源
     - 策略版本
     - 纪律检查结果
     - 仓位风险摘要
     - 预计止盈止损结构

### 关键原则
- 系统内部先生成 `OrderIntent`，通过纪律和风控引擎审核后，才变成真实 `Order`
- 执行适配器与策略/Agent 解耦，避免“策略逻辑直接下单”

---

## 6.3 纪律执行模块（产品核心）

这是悟道最重要的差异化能力之一。

### 目标
将“纪律”从口头原则变成软件可执行规则。

### 功能
1. **交易前纪律检查**
   - 是否有明确交易计划
   - 是否填写入场理由
   - 是否设置止损
   - 是否超过单笔风险上限
   - 是否在禁止交易时段内
   - 是否违反当日最大亏损限制
   - 是否违反连续亏损冷静期规则
   - 是否属于未审批策略

2. **交易中纪律监控**
   - 是否擅自扩大亏损
   - 是否撤掉原定止损
   - 是否超计划加仓
   - 是否逆势频繁补仓
   - 是否违反预定分批止盈/止损计划

3. **交易后纪律审查**
   - 本笔交易是否严格按照计划执行
   - 偏差发生在哪个步骤
   - 属于策略问题、执行问题、心理问题还是纪律问题

4. **纪律提示面板**
   - Jason 自定义提示语
   - 关键纪律提醒
   - 当天风险状态
   - 当前最容易犯的错误提示

### 纪律规则示例
- 单笔风险不超过账户净值的 X%
- 同方向连续加仓次数不超过 N 次
- 当日最大亏损达到阈值后禁止继续开新仓
- 非计划内交易必须经过额外确认
- 某些策略只允许在指定市场环境下启用

### 产出
每笔交易都能留下：
- 哪些规则通过
- 哪些规则失败
- 是否人工 override
- override 的理由与责任归属

---

## 6.4 持仓与风险管理模块

### 目标
把“仓位管理”和“止盈止损管理”从券商原始能力提升到策略级别与组合级别。

### 功能
- 当前持仓总览
- 按策略 / 标的 / 账户 / 市场分类展示
- 实时盈亏、浮动风险、风险暴露
- 高级止盈止损：
  - 固定止损
  - 移动止损
  - 保本线
  - 分批止盈
  - 达到目标后只留 runner
  - 时间止损
- 仓位管理：
  - 单笔仓位上限
  - 单策略总风险预算
  - 总账户风险预算
  - 相关性集中度限制

### 核心特点
悟道不应只显示“你持有什么”，而应显示：
- 你为何持有
- 哪个策略持有
- 当前风险是否合规
- 下一步计划是什么

---

## 6.5 策略管理模块

### 目标
统一管理不同类型、不同市场、不同逻辑来源的策略。

### 策略来源
1. Jason 手动提出
2. AI Agent 提案
3. 外部平台信号（TradingView / 数据服务）
4. 基于新闻、链上、宏观、套利等来源形成的策略

### 策略信号类型
悟道的策略层应明确兼容三类信号来源：
1. **规则型策略**：技术指标、入场/出场规则、风险约束
2. **数据驱动策略**：因子、分类/回归/排序模型、特征工程结果
3. **AI / Agent 建议型策略**：AI 给出候选方向，但必须结构化落地为可解释信号

这意味着策略管理模块管理的不是“某一种模型”，而是统一的策略资产与信号资产。

### 策略分类
1. **宏观与新闻策略**
2. **链上追踪策略**
3. **数据驱动与套利策略**
4. **网格与做市类策略**
5. **专项平台策略**（如 Polymarket）
6. **人工 discretionary + AI 辅助策略**

### 策略对象模型建议
每个策略应至少包含：
- 策略名称
- 策略类型
- 描述
- 版本号
- 作者（Jason / AI Agent / 联合）
- 参数集
- 适用市场
- 适用标的池
- 启用状态
- 风险等级
- 绩效指标
- 关联回测结果
- 关联纪律约束

### 策略生命周期
- Draft
- Review
- Simulated
- Approved
- Paper / Shadow
- Live
- Paused
- Retired

### 为什么这块重要
你不是只需要“一个策略列表”，而是需要一个**统一的策略资产管理器**。

## 6.6 回测、验证与研究模块

### 目标
用成熟工具 + 系统内统一记录的方式，对策略做可比较、可追溯的验证。

### 功能
- 简单回测
- 参数回测
- 不同市场环境下分段测试
- 交叉验证 / Walk-forward 思路支持
- 基准对比
- 回测结果版本化保存
- 关联到策略版本
- 策略评估中心（Strategy Evaluation Center）
- 评估结果快照沉淀与比较
- Paper / Shadow Mode 跟踪

### 策略评估中心应至少展示
- 累计收益率
- 年化收益率
- 夏普比率
- 最大回撤
- 胜率
- 盈亏比
- 交易次数
- 平均持仓时长
- 分市场环境表现

研究模块不应只有一个 backtest 按钮，而应成为真正的**策略决策面板**。

### 建议
- 第一阶段不从零造复杂回测平台
- 优先接外部成熟工具（TradingView、vectorbt、Backtrader 等）
- 悟道负责：
  - 回测任务编排
  - 回测结果管理
  - 策略版本关联
  - 结果比较与展示

### 结合现有原型能力
当前已有：
- `backtest.py`：简化版规则回测器

可复用为：
- MVP 级研究验证入口

但不足：
- 无手续费、滑点、冲击成本
- 无多策略/多标的组合回测
- 无参数管理和实验管理
- 仅适合作为原型，不适合作为最终研究基础设施

## 6.7 交易记录与复盘模块

### 目标
完整记录交易行为，并支持纪律复盘、策略复盘与心理复盘。

### 记录对象
- 原始信号
- 下单意图
- 订单状态
- 成交记录
- 仓位变化
- 策略版本
- 纪律检查结果
- AI 建议与解释
- Jason 手写备注

### 复盘维度
- 这笔交易赚/亏了多少
- 是否符合策略
- 是否符合纪律
- 偏差来自哪里
- 是否应该继续使用该策略
- 是否需要调整风控参数

### 存储建议
交易记录要有两层：
1. **结构化存储**：数据库 / 本地状态文件
2. **可读复盘层**：Notion / Markdown / 周报

### 结合现有原型能力
当前已有：
- `journal.py`：本地 trade journal
- 初步 Notion sync 能力

可复用为：
- MVP 级交易日志与摘要模块

不足：
- 还没有完整订单/成交/持仓事件模型
- Notion sync 较粗糙，需要重做成正式同步服务

## 6.8 提醒与通知模块

### 目标
把“价格提醒”升级为“交易事件提醒中心”。

### 提醒类型
- 价格触发
- 波动异常
- 纪律违规
- 订单状态更新
- 止盈止损触发
- 策略状态异常
- AI Agent 提醒（研究结论 / 风险提示 / 需要审批）

### 通知渠道
- 终端内提醒
- Telegram
- 未来可扩邮件 / 移动推送

### 结合现有原型能力
当前已有：
- `alerts.py`：阈值提醒
- `notify.py`：Telegram 通知原型
- `check-alerts-cron`：定时检查脚本

可复用为：
- MVP 级提醒系统骨架

不足：
- 通知通道实现较脆弱
- 缺少正式的事件总线和通知中心
- 缺少告警分级、聚合、抑制、恢复通知

## 6.9 AI Agent 协同模块

### 目标
让 AI Agent 成为交易系统中的“第二大脑”，但始终在明确权限边界下工作。

### AI 可以做的事
- 研究任务拆解
- 策略建议
- 参考资料搜索
- 行情或风险解释
- 复盘总结
- 交易纪律提醒
- 策略版本说明
- 回测结果解读

### AI 不应默认做的事
- 自动创建真实订单
- 自动启用新策略
- 自动修改关键风控参数
- 静默推进真实交易行为
- 在后台悄悄改变系统状态而不告知 Jason

### 推荐的 Agent 模式
1. **Advisory Mode（默认）**
   - 只做建议，不执行
2. **Semi-Auto Mode**
   - AI 可生成订单意图，但必须人工确认
3. **Auto Mode（未来）**
   - 仅限特定策略、特定账户、特定额度、特定风控护栏下启用

- 首页 / 总览
- 行情
- 持仓
- 执行
- 策略
- 回测与研究
- Journal / 复盘
- Agent 建议
- 设置

建议默认展示：
- 自定义提醒语 / 纪律提示
- 当前持仓与风险

## 8. 数据模型建议

建议至少定义以下核心对象：

- `Instrument`：标的
- `Watchlist`：观察列表
- `SignalArtifact`：研究/策略侧产出的标准信号产物
- `Signal`：系统内部信号
- `OrderIntent`：待执行订单意图
- `Order`：订单
- `Fill`：成交
- `Position`：持仓
- `Strategy`：策略
- `StrategyVersion`：策略版本
- `EvaluationSnapshot`：策略评估结果快照
- `RiskRule`：风控规则
- `DisciplineRule`：纪律规则
- `Alert`：提醒
- `JournalEntry`：复盘条目
- `AgentSuggestion`：AI 建议
- `BacktestRun`：回测任务
- `BacktestResult`：回测结果

这些对象之间应具备清晰关联，例如：
- 一条 `SignalArtifact` 如何转化为系统内部 `Signal`
- 一笔订单来自哪个信号
- 某信号属于哪个策略版本
- 某个 `EvaluationSnapshot` 对应哪个策略版本与时间区间
- 某持仓由哪些成交组成
- 某交易被哪些纪律规则放行
- 某 AI 建议是否被采纳

## 10. 推荐的产品迭代路线

## Phase 0：整理现有原型（立即可做）
目标：把已有 trading-agent 原型整理成“可复用模块”，而不是继续无序堆功能。

输出：
- 模块清单
- 可复用 / 需重写边界
- 技术债清单

## Phase 1：悟道 MVP Alpha（个人研究 + 纪律终端）
目标：先做“每天愿意打开”的个人交易终端。

范围：
- 行情面板
- 持仓总览
- 纪律提示区
- 交易日志
- AI 建议区
- 简单策略管理
- 提醒中心

此阶段**不追求自动执行**。

## Phase 2：悟道 MVP Beta（评估与影子运行）
目标：在不触发真实执行复杂度爆炸的前提下，让策略闭环更完整。

范围：
- 策略评估中心
- `EvaluationSnapshot` 管理
- `Paper / Shadow Mode`
- Signal -> OrderIntent 的标准化衔接
- 策略比较与研究结果沉淀

## Phase 3：半自动交易执行
目标：在纪律与风控充分成形后，接入 TradingView webhook 与部分券商执行。

范围：
- Signal -> OrderIntent -> Discipline Check -> Manual Confirm -> Execution
- 高级止盈止损
- 订单状态回传

## Phase 4：策略工作台与验证体系
目标：把策略真正纳入版本控制、回测、对比、审批流程。

范围：
- 策略版本管理
- 参数管理
- 回测结果管理
- 实盘/回测对比
- 策略审批流

## Phase 5：受控自动化
目标：只对少数成熟策略开放有限自动执行能力。

范围：
- 策略级授权
- 额度限制
- 风险熔断
- 自动化行为全量审计

## 11. MVP 成功标准

如果悟道 v2 的第一阶段做得对，应该达到以下结果：

1. Jason 每天愿意打开它看盘与复盘。
2. 每一笔真实交易都能在系统中被记录与解释。
3. 交易前能看到纪律检查结果，而不是靠脑子硬记。
4. AI Agent 提供的建议是有帮助的，但不会偷偷越权执行。
5. 当前持仓、策略状态、风险状态一眼能看清。
6. 未来要接自动化时，不需要推翻整套架构。

## 12. 最终结论

悟道的正确方向不是“再造一个券商 App”，而是：

**为 Jason 构建一个以纪律、风控、人机协同和策略管理为核心的个人交易操作系统。**

当前已有的 trading-agent 原型，已经覆盖了以下几个非常有价值的起点：
- 行情数据
- 基础看板
- 技术分析
- 提醒
- 交易日志
- 简单回测

它们不是最终产品，但足够成为悟道 v2 的底层原型输入。

下一步最合理的产品策略，不是继续零散叠功能，而是把它们收束到统一产品架构中：

**悟道 = 交易终端 + 纪律引擎 + 策略管理器 + AI 工作台。**

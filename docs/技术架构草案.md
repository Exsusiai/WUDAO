# 悟道 - 技术架构草案

> 本文基于《产品设计-v2.md》与《MVP功能与阶段路线.md》整理。
> 目标：把“悟道”从产品概念进一步压缩成一个可开发、可演进、复杂度可控的技术架构。

---

## 1. 架构目标

悟道的技术架构必须同时满足 5 个目标：

1. **先把 MVP 做出来**
   - 支持看盘、持仓、纪律、记录、AI 建议
   - 不因过早引入真实执行而爆炸

2. **天然支持后续扩展**
   - 未来可接 TradingView webhook
   - 可接券商/交易所
   - 可接更复杂的策略与回测

3. **AI 可接入，但边界清晰**
   - AI 默认只能产出建议、分析、总结
   - 不能默默写入关键交易状态
   - 更不能默认直接获得交易执行权限

4. **状态可追溯**
   - 每一条交易建议、纪律检查、订单意图、日志记录、策略修改都要可审计

5. **尽量复用现有原型**
   - `~/projects/trading-agent/` 不是最终产品，但应作为模块原型仓库使用

---

## 2. 研究到执行的高层流水线

从业务闭环视角，悟道应围绕下面四层流水线展开：

1. **Market Data Layer**
   - 行情接入、历史数据、watchlist、特征原始输入
2. **Strategy / Signal Layer**
   - 规则策略、数据驱动策略、AI 候选建议、信号生成
3. **Research / Evaluation Layer**
   - 回测、参数实验、绩效比较、评估结果沉淀
4. **Execution / Serving Layer**
   - `Signal` / `OrderIntent` 消费、纪律检查、人工确认、真实执行

这四层不是部署单元，而是系统边界设计原则：
- 数据层不要和执行逻辑耦死
- 研究层不要直接下单
- 执行层不要直接依赖训练/研究代码内部实现
- 中间要靠标准化对象衔接

## 3. 核心架构结论

我建议悟道采用：

**前后端分离的本地优先 Web 架构**

而不是：
- 纯 Streamlit 单体
- 纯脚本堆叠
- 一开始就做原生桌面端
- 一开始就做分布式微服务

### 推荐形态
- **前端**：React / Next.js
- **后端 API**：Python FastAPI
- **后台任务/调度**：Python Worker（与 API 分进程）
- **数据库**：SQLite（MVP），后续可升级 PostgreSQL
- **数据/策略/分析模块**：优先用 Python，复用现有原型能力
- **AI 集成**：单独的 Agent Bridge，默认 advisory only

### 为什么这么选

#### 选 React / Next.js 做前端
因为悟道最终不是一个简单报表页，而是一个多面板工作台：
- 总览
- 行情
- 持仓
- 纪律检查
- 策略管理
- 复盘
- AI 建议

这种交互复杂度，Streamlit 迟早撑不住。

#### 选 FastAPI 做后端
因为你现有 trading-agent 原型几乎全是 Python：
- 数据抓取
- 技术分析
- 日志
- 回测
- 提醒

直接用 Python 后端能最大限度复用。

#### 选 SQLite 做 MVP 数据库
因为当前是单人本地工具，SQLite 的优点太明显：
- 简单
- 部署轻
- 备份方便
- 易于迭代

等未来真的需要：
- 多进程并发更强
- 更复杂查询
- 远程部署
再升 PostgreSQL。

#### 先做 Web，不先做原生桌面
因为你现在真正需要的是：
- 快速迭代
- UI 可塑性
- 容易接 API
- 容易调试

不是打包一个桌面壳子来增加复杂度。

## 4. 高层模块划分

建议将悟道拆成 6 个技术模块。

### 4.1 Frontend App
职责：
- 页面路由
- 面板布局
- 表单交互
- 图表展示
- 状态管理
- 审批/确认交互

对应页面：
- Dashboard
- Watchlist
- Instruments
- Positions
- Journal
- Strategies
- Discipline
- Agent Suggestions
- Settings

---

### 4.2 API Service
职责：
- 对外提供统一业务接口
- 聚合数据
- 提供 CRUD
- 调用纪律引擎
- 提供策略管理接口
- 提供 AI 建议查询/提交接口
- 提供回测任务与结果读取接口

API 是整个系统的主编排层。

---

### 4.3 Worker / Scheduler
职责：
- 周期性拉取市场数据
- 检查提醒
- 生成日报/摘要
- 执行轻量后台任务
- 将来接 TradingView webhook 后处理异步工作

MVP 阶段不要引入复杂消息队列。
先用：
- 单独 worker 进程
- 定时任务
- 数据库 job 表

就够了。

---

### 4.4 Domain Modules
这是核心业务层，建议按能力拆：
- market
- portfolio
- discipline
- journal
- strategy
- alerts
- research
- agent
- execution（未来）

这些模块不要直接写死在页面里，也不要让页面直接拼业务规则。

---

### 4.5 Adapters
负责接外部世界：
- yfinance
- ccxt
- TradingView webhook（未来）
- broker APIs（未来）
- Telegram/消息通知
- Notion（可选）
- OpenClaw / AI bridge

原则：
**外部依赖都放在 adapter 层，不让它们污染核心领域模型。**

---

### 4.6 Storage Layer
职责：
- 结构化持久化
- 审计日志
- 历史记录
- 配置存储

MVP 先统一放 SQLite。
如果有少量配置文件，也只能作为补充，不要让 JSON 文件再次野蛮生长成新的状态地狱。

## 5. 推荐的目录结构

建议新项目采用 polyglot monorepo：

```text
wudao/
├── apps/
│   └── web/                  # Next.js 前端
├── services/
│   ├── api/                  # FastAPI 主服务
│   └── worker/               # 后台任务/调度
├── packages/
│   ├── domain-docs/          # 数据模型、接口协议、ADR 等文档
│   ├── ui/                   # 可复用前端组件（可选）
│   └── schemas/              # OpenAPI / JSON Schema / 共享 DTO
├── python/
│   ├── market/               # 行情与数据适配模块
│   ├── analysis/             # 技术分析模块
│   ├── journal/              # 交易日志模块
│   ├── discipline/           # 纪律引擎
│   ├── strategy/             # 策略管理领域逻辑
│   ├── alerts/               # 提醒逻辑
│   ├── research/             # 回测与研究逻辑
│   ├── agent_bridge/         # AI/Agent 适配
│   └── execution/            # 执行适配（未来）
├── infra/
│   ├── db/
│   │   ├── migrations/
│   │   └── seed/
│   └── scripts/
├── docs/
└── README.md
```

### 为什么不建议继续把所有逻辑堆在一个脚本目录里
因为你已经能看到原型的问题了：
- 原型快，但边界糊
- 很多功能是“能跑”而不是“能演进”
- 通知、提醒、日志、分析、CLI 都搅在一起

如果悟道继续沿着这个方式长，会很快变成维护灾难。

## 6. MVP 推荐技术选型

## 6.1 前端
推荐：
- **Next.js + TypeScript + Tailwind + shadcn/ui**

理由：
- 面板类产品开发效率高
- 组件生态成熟
- TypeScript 约束好
- 后续可接桌面壳（如 Tauri）也不难

MVP 不必一开始就搞：
- Zustand / Redux 重度状态系统
- 复杂 SSR 优化
- 微前端

前端第一阶段只要：
- 路由清晰
- 数据获取稳定
- 表单/表格/卡片体验好

---

## 6.2 后端
推荐：
- **FastAPI + SQLAlchemy / SQLModel + Pydantic**

理由：
- 与现有 Python 原型对接自然
- 数据分析逻辑直接复用
- API 定义与 schema 清楚
- 后续回测/研究/适配器也都适合 Python

---

## 6.3 数据库
MVP：
- **SQLite**

后续升级：
- **PostgreSQL**

升级条件建议：
- 需要远程部署
- 需要更强并发
- 需要更复杂任务调度
- 需要更多历史分析查询

---

## 6.4 调度/后台任务
MVP：
- 独立 Python worker
- APScheduler 或简易 cron + jobs table

不建议 MVP 一开始就引入：
- Celery
- Kafka
- RabbitMQ
- 一堆你将来只会骂自己的基础设施

---

## 6.5 图表
建议：
- 前端使用成熟图表库
- 不自建图表引擎

MVP 优先级：
- K 线基础展示
- 价格线图
- 指标叠加
- P&L 趋势

## 10. 关键数据流

## 10.1 观察流
1. Worker 拉取行情
2. 写入 market cache / snapshot
3. API 聚合后返回给 Dashboard
4. 前端展示 watchlist 与详情页

## 10.2 交易计划流
1. Jason 在前端创建 `TradePlan`
2. API 调用 Discipline Engine
3. 返回检查结果：通过 / 警告 / 不通过
4. 保存检查记录与原因
5. Jason 决定是否继续

## 10.3 研究到执行衔接流
1. Strategy / Research 层产出 `SignalArtifact`
2. API / Worker 校验其 schema、来源与策略版本
3. 将其转化为系统内部 `Signal` 或 `OrderIntent`
4. 进入 Discipline Engine 与审批链路
5. 通过后进入 `Paper / Shadow Mode` 或真实执行
6. 结果回写为 `EvaluationSnapshot`、`ShadowRun` 或执行事件

## 10.4 交易记录流
1. Jason 手动录入交易
2. API 写入 `trade_records`
3. Portfolio 模块更新持仓聚合
4. Journal 模块生成复盘入口
5. Dashboard 刷新摘要

## 10.5 AI 建议流
1. Jason 在某个策略/标的/交易记录页面触发 AI 建议
2. API 调用 Agent Bridge
3. Agent 输出结构化结果：
   - task
   - suggestions
   - references
   - risks
4. 保存为 `agent_suggestions`
5. 页面展示结果

### 关键限制
- AI 输出先存为建议对象
- 不直接写策略、持仓、纪律状态
- 任何状态变更都必须通过普通业务接口显式执行

## 12. AI / Agent 集成边界

这是悟道里非常重要的一层，必须提前钉死。

## 12.1 默认模式：Advisory Only
AI 可以：
- 给建议
- 做解释
- 找参考资料
- 给风险提示
- 做复盘总结

AI 不可以默认：
- 写订单
- 改风控参数
- 改策略状态
- 改持仓
- 改纪律结果

## 12.2 推荐的 Agent Bridge 设计
API 不直接依赖某个具体大模型，而是通过一个 `Agent Bridge`：
- 输入：上下文 + 任务类型 + 结构化请求
- 输出：结构化建议结果

建议支持的任务类型：
- `instrument_review`
- `strategy_review`
- `trade_review`
- `reference_search`
- `risk_commentary`

## 12.3 所有 AI 输出都要落库
至少保存：
- prompt 摘要
- 任务类型
- 输入关联对象
- suggestions
- references
- risks
- timestamp

这样以后：
- 能复查 AI 到底说了什么
- 能评估 AI 建议是否有用
- 不会变成“聊天里说过但系统里没记录”的黑箱

## 13. 安全与权限建议

即便现在是个人产品，也要早点把边界立住。

### 12.1 凭证隔离
- 券商/交易所密钥不要暴露给 AI 层
- 执行适配器的凭证只放在 execution adapter 层
- 前端永远不直接接触敏感密钥

### 12.2 关键动作审批
将来如果启用执行能力，至少这些动作要确认：
- 新策略启用
- 策略参数大改
- 自动化模式开关
- 下真实订单

### 12.3 审计日志保留
所有关键操作写 `audit_logs`。

---

## 14. MVP 部署形态

建议 MVP 采用：

### 本地单机部署
- 前端：localhost Web
- API：本机 FastAPI
- Worker：本机后台进程
- DB：本地 SQLite 文件

### 为什么这样最好
- 开发成本低
- 状态容易掌控
- 调试简单
- 适合 Jason 个人使用

等真正证明每天都在用，再考虑：
- 局域网访问
- 云端部署
- 桌面壳封装

## 15. 推荐的第一批开发顺序

不是所有模块并行开工。

### Step 1：搭项目骨架
- monorepo 结构
- web + api + db 基础连通
- 基础页面框架

### Step 2：先做 Dashboard / Watchlist / Journal
因为这是最容易形成“每天愿意打开”的部分。

### Step 3：加入 Discipline Engine v1
这是 MVP 的产品灵魂。

### Step 4：加入 Strategy Manager v1
把策略收口，不再四处漂浮。

### Step 5：接 AI Suggestion 面板
让 AI 真正进产品，而不是停留在聊天里。

### Step 6：最后再补提醒与回测入口
先做骨架和闭环，再做增强件。

## 16. 一句话总结

悟道最合理的技术路线不是：

**“继续堆一个更大的 trading-agent 脚本集合”**

而是：

**用 React/Next.js + FastAPI + SQLite 做一个本地优先的个人交易工作台，**
**把市场观察、持仓、纪律、日志、策略、AI 建议收进统一架构，**
**并为未来的执行能力预留边界，但不提前引爆复杂度。**
